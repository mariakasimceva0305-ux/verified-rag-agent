# Verified RAG Agent

Воспроизводимый локальный контур ответа по документам: переформулировка запроса (эвристики, без внешних LLM), векторный поиск, лексическое переранжирование, извлечение ответа из контекста с цитатами и проверка согласованности с чанками по пересечению формулировок. В репозитории — демо-корпус, конфигурации `baseline` / `improved` и сценарии `ingest`, `build_index`, `run_eval`.

## Назначение

Снизить долю ответов, слабо опирающихся на найденный контекст, и сделать выдачу проверяемой за счёт ссылок на чанки и отдельного этапа согласования с источниками.

## Архитектура

```text
Вопрос
  │
  ▼
Переформулировка запроса
  │
  ▼
Поиск документов
  │
  ▼
Переранжирование
  │
  ▼
Генерация ответа
  │
  ▼
Проверка ответа по источникам
  │
  ▼
Ответ с цитированием
```

## Что реализовано

- переформулировка запроса перед поиском;
- поиск и переранжирование найденных документов;
- формирование ответа только по найденному контексту;
- цитирование использованных источников;
- отдельный этап проверки ответа;
- материалы и сценарии оценки.

## Структура репозитория

- `configs/` — конфигурации запуска;
- `data/` — данные и демонстрационные материалы;
- `docs/` — пояснения по архитектуре;
- `eval/` — сценарии и артефакты оценки;
- `examples/` — примеры использования;
- `scripts/` — сценарии запуска;
- `src/` — основной код;
- `tests/` — тесты.

## Оценка качества

Основные метрики:

- `HitRate@K`
- `MRR`
- доля ответов, подтверждённых найденными источниками (при наличии отдельного расчёта в сценарии оценки)
- задержка ответа: `p50`, `p95`

Сравнения по контурам:

- поиск без переформулировки и переранжирования (`configs/baseline.yaml`);
- поиск с переформулировкой и переранжированием и этапом проверки (`configs/improved.yaml`).

## Результаты

Оценка: `python scripts/run_eval.py` после `ingest` и `build_index` для выбранной конфигурации, набор `eval/questions.jsonl` (16 вопросов). Запуск из корня репозитория с `PYTHONPATH=.` (или эквивалентом). Итоги дублируются в `eval/results/summary_<stem>.json` и построчно в `eval/results/runs_<stem>.jsonl`.

**Как считается.** Для каждого вопроса строится запрос к индексу; `HitRate@K` — есть ли эталонный документ среди top‑K после ретрива (до переранжирования, чтобы сравнение режимов было по одному правилу). `MRR` — по позиции первого совпадения с эталоном в финальной выдаче. На Windows пути в `gold_doc_ids` могут отличаться от `doc_id` в индексе; скрипт оценки нормализует это через алиасы путей, не меняя `eval/questions.jsonl`. В контуре `improved` дополнительно считается доля ответов, прошедших проверку по чанкам (`grounded_rate`); в `baseline` верификатор выключен, поэтому поле в сводке отсутствует.

| Конфигурация | HitRate@K | MRR | grounded_rate | latency_p50_ms | latency_p95_ms |
|--------------|-----------|-----|---------------|----------------|----------------|
| `configs/baseline.yaml` | 1.0 | 0.9375 | — | 6.52 | 8.07 |
| `configs/improved.yaml` | 1.0 | 0.9688 | 1.0 | 7.25 | 9.24 |

Таблица отражает последний локальный прогон на демо-корпусе и индексах из `data/processed/`; при смене корпуса или модели эмбеддингов метрики получают тем же сценарием `run_eval`.

## Запуск

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# пример: подготовка и оценка для одной конфигурации
export PYTHONPATH=.
python scripts/ingest.py --config configs/baseline.yaml
python scripts/build_index.py --config configs/baseline.yaml
python scripts/run_eval.py --config configs/baseline.yaml

python scripts/demo.py --config configs/improved.yaml
```

В среде Windows перед запуском скриптов задают переменную окружения `PYTHONPATH` на корень репозитория (аналог `export PYTHONPATH=.`).

## Ограничения

- качество ответа определяется полнотой корпуса и качеством поиска;
- дополнительные этапы повышают задержку;
- переформулировка, переранжирование и верификатор — эвристики; они не заменяют независимую экспертную или LLM‑оценку фактов;
- для стабильного сравнения режимов сохраняют отдельные отчёты по каждому сценарию оценки.
